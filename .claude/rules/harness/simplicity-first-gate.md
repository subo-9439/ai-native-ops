# 간결 구현 게이트 (simplicity-first-gate)

> **사용자 지시 (하네스 전문가 경유, 2026-05-19)**: GitHub 109K★ Karpathy
> CLAUDE.md 원칙2 — "Solve in the simplest way that fully works. Less code is
> better. Don't add features I didn't ask for. Avoid unnecessary abstraction."
> 우리 14게이트엔 부재 → 신규 게이트화.
>
> 위반 형태: 200줄로 끝날 작업을 추상 레이어/제네릭/플래그 잔뜩 얹어 600줄로
> 작성. 요청하지 않은 "있으면 좋을" 옵션·설정·헬퍼를 추측해서 추가. 시니어
> 엔지니어가 보면 "이건 과설계"라 부를 코드.

## 원칙

**완전히 동작하는 가장 단순한 방법으로 푼다. 코드는 적을수록 좋다.**
요청하지 않은 기능을 추측으로 추가하지 않는다. 불필요한 추상화를 만들지 않는다.

"200줄로 될 걸 50줄로." 시니어 엔지니어가 과설계라 부를 코드를 제거한다.

## 강제 규칙

| # | 규칙 |
|---|------|
| 1 | **최소 코드**. 같은 요구를 더 적은 줄로 충족할 방법이 있으면 그쪽을 택한다. 줄 수 늘리는 변경은 그 줄이 요청에 직접 필요한 경우에만. |
| 2 | **추측 기능 0**. 사용자가 명시하지 않은 옵션·플래그·설정·확장 포인트·"미래 대비" 코드 추가 금지. 필요해질 때 그때 추가한다 (YAGNI). |
| 3 | **불필요 추상화 제거**. 호출처가 1곳인 헬퍼/래퍼/제네릭/인터페이스/추상 클래스 신설 금지. 중복이 실제 3회 이상 발생한 뒤에만 추출한다. |
| 4 | **과설계 금지**. 디자인 패턴·레이어·이벤트 버스·DI 컨테이너를 요구 규모에 안 맞게 도입 금지. 시니어가 "이거 왜 이렇게 복잡해?" 물을 코드면 다시 줄인다. |
| 5 | **직선 코드 우선**. 조기 일반화보다 명시적·직선적 코드. 분기 3개를 전략 패턴으로 바꾸지 말 것. |
| 6 | 단순화가 요구를 일부라도 깨면 단순화하지 않는다. "완전히 동작" 이 1순위, "가장 단순" 이 그 다음. |

## 적용 대상

- 모든 코드 작성/수정 PR (Flutter / Spring BE / AI / 스크립트 / hook)
- 신규 클래스·함수·모듈·엔드포인트 추가
- 리팩터링 PR (단순화 방향만 인정, 복잡화 금지)

## 적용 제외

- 문서 / 메모리 / 핸드오버 / plan.md 수정
- 긴급 hotfix (사유를 답변에 명시: `간결 검토 생략: hotfix <사유>`)
- 사용자가 명시적으로 요청한 추상화·확장 포인트 (요청 자체가 근거)

## 답변 검증 표현

코드 작성/수정 PR 답변에 **다음 한 줄 필수** (마지막 줄 근처):

- ✅ `간결 구현 검증: 최소범위 ✓ / 추측기능 0 ✓ / 과설계 0 ✓`
- ⚠️ `간결 검토 생략: hotfix <사유>` (긴급 예외 시)

명시가 없으면 본 게이트를 무시한 것으로 간주한다.

## 다른 게이트와의 구분

| 게이트 | 축 |
|--------|----|
| **simplicity-first** (이 파일) | 변경 **방식** 최소화 — 적은 코드 / 추측기능 0 / 과설계 0 |
| `surgical-change` | 변경 **범위** 최소화 — 요청한 것만, 인접 코드 안 건드림 |
| `polish-pass` | UI/UX 완료 기준 — DoD 7항목 |
| `connected-work` | 데이터 흐름 6단계 점검 |
| `concise-response` | 답변 **길이** 최소화 (코드 아님) |

simplicity 는 "어떻게 적게 짤까", surgical 은 "어디까지만 건드릴까" — 짝 게이트.
