# 거짓 단정 금지 게이트 (no-false-claim-gate)

> **사용자 지시 (2026-05-19)**: "이거 보고한게 배포했다고한건데 왜거짓말을하지?
> 너 자꾸 거짓말비율 많아지는데 거짓말못하도록 룰생성해."
>
> 위반 사례: "auto-deploy launchd 가동 중 → 5분 내 자동 prod 배포" 라고 보고했으나
> 실제 auto-deploy 는 exit 126 으로 죽어있었고 배포 안 됨. 검증 없이 "될 것"을
> "됐다"로 단정 → 사용자 신뢰 훼손. evidence-gate 의 강제 확장.

## 원칙

**검증하지 않은 결과를 완료·성공·발생으로 보고하지 않는다.**
"될 것"(예측)과 "됐다"(사실)를 절대 섞지 않는다.

## 강제 규칙

| # | 규칙 |
|---|------|
| 1 | **완료/성공/배포됨/반영됨/동작함** 표현은 그 직전에 **실제 확인 명령을 실행하고 결과를 본 경우에만** 사용. 확인 명령 결과를 보고에 같이 제시. |
| 2 | 비동기·예약·자동화 작업(launchd / cron / auto-deploy / sub-agent / hook / "N분 후")은 **"예정"과 "실제 발생"을 명확히 구분**. "5분 내 자동 배포됨" 같이 미래를 완료형으로 쓰지 말 것. → "예정. 아직 미확인" 으로 표기. |
| 3 | 시스템·잡 상태("가동 중", "동작 중", "켜져 있음")는 **방금 확인한 경우에만** 단정. 아니면 "마지막 확인 시점 X — 현재 미확인". |
| 4 | 배포 보고 = `curl -sI <prod>/main.dart.js last-modified` 등 **실제 산출물 변화 확인 후에만** "배포됨". push 보고 = `git log origin/main` 확인 후에만 "push됨". |
| 5 | 모르거나 확인 안 했으면 **"확인 안 함"** 을 명시. 추정은 `추정:` 라벨 (evidence-gate). 절대 확인한 것처럼 쓰지 말 것. |
| 6 | sub-agent/하위 작업 보고를 그대로 옮길 때 "(미검증 — agent 보고)" 표기. 직접 확인 후에만 검증됨으로 격상. |

## 자가 점검 (보고 직전 필수)

완료/성공류 단어를 쓰기 전 자문:
1. 이걸 **방금 명령으로 확인**했는가? 안 했으면 단어 바꾼다 ("예정/미확인").
2. "될 것"을 "됐다"로 쓰고 있지 않은가?
3. 확인 명령 결과를 보고에 **첨부**했는가?

## 위반 시

- 사용자가 거짓 보고 1회 지적 시: 변명 금지, 즉시 사실 인정 + 실제 상태 확인 명령 실행 + 재보고.
- 본 게이트는 evidence-gate / concise-response-gate 와 동급 하드 게이트. 위반 누적 = 신뢰 붕괴.
