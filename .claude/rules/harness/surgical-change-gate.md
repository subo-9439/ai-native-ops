# 수술적 변경 게이트 (surgical-change-gate)

> **사용자 지시 (하네스 전문가 경유, 2026-05-19)**: GitHub 109K★ Karpathy
> CLAUDE.md 원칙3 — "Change only what I asked. Don't 'improve' adjacent code,
> comments, or formatting. Don't refactor code that isn't broken. Match the
> existing style even if it's suboptimal. Only remove orphan imports/vars your
> change created." 우리 14게이트엔 부재 → 신규 게이트화.
>
> 위반 형태: 한 함수 fix 요청에 옆 함수까지 "더 좋게" 리네임/리포맷/리팩터.
> 안 깨진 코드를 임의로 갈아엎음. 본인이 만들지 않은 기존 dead code 를 명시
> 요청 없이 제거. 변경 라인이 사용자 요청에 추적되지 않음.

## 원칙

**요청한 것만 수정한다.** 인접 코드·주석·포맷을 "개선" 하지 않는다.
안 깨진 코드를 리팩터하지 않는다. 기존 스타일을 (suboptimal 이어도) 매칭한다.

모든 변경 라인은 사용자 요청에 **직접 추적** 가능해야 한다.

## 강제 규칙

| # | 규칙 |
|---|------|
| 1 | **요청 범위만**. 사용자가 지시한 함수/파일/동작만 변경. 옆 함수·다른 파일은 같은 PR 에 필요하다는 직접 근거(데이터 흐름·빌드 의존)가 없으면 안 건드린다. |
| 2 | **인접 개선 0**. 주변 코드의 네이밍·주석·포맷·import 정렬·들여쓰기를 "겸사겸사" 고치지 않는다. 보기 거슬려도 그대로 둔다. |
| 3 | **안 깨진 코드 리팩터 금지**. 동작하는 코드를 더 나아 보인다는 이유로 재구조화 금지. 리팩터는 사용자가 명시 요청한 별도 PR 에서만. |
| 4 | **기존 스타일 매칭**. 파일의 기존 네이밍·따옴표·포맷 컨벤션이 suboptimal 이어도 그 파일 안에서는 그대로 따른다. 프로젝트 스타일을 PR 에 섞어 바꾸지 않는다. |
| 5 | **orphan 정리 한정**. 본인 변경으로 생긴 미사용 import/변수만 제거. 기존부터 있던 dead code 는 명시 요청 없으면 손대지 않는다 (simplicity-first 와 충돌 시 본 규칙 우선 — 범위 보호). |
| 6 | **추적성**. 모든 변경 hunk 가 "사용자가 X 요청 → 이 줄" 로 설명돼야 한다. 설명 안 되는 hunk 는 PR 에서 뺀다. |

## 적용 대상

- 모든 코드 작성/수정 PR (Flutter / Spring BE / AI / 스크립트 / hook)
- 버그 fix, 기능 추가, 작은 edit 전부

## 적용 제외

- 문서 / 메모리 / 핸드오버 / plan.md 수정
- 긴급 hotfix (사유를 답변에 명시)
- 사용자가 명시적으로 "리팩터해줘 / 포맷 정리해줘 / dead code 치워줘" 요청한 PR (요청 자체가 범위 근거)

## 답변 검증 표현

코드 작성/수정 PR 답변에 **다음 한 줄 필수**:

- ✅ `수술적 변경 검증: 요청범위만 ✓ / 인접개선 0 ✓ / 스타일매칭 ✓`
- ⚠️ `수술적 검토 생략: <리팩터 PR / hotfix 사유>` (예외 시)

명시가 없으면 본 게이트를 무시한 것으로 간주한다.

## connected-work-gate 와의 구분 (중요)

surgical 은 "범위를 좁혀라", connected 는 "데이터 흐름은 끝까지 따라가라" — 반대 축이라 헷갈리기 쉽다.

| 항목 | surgical-change | connected-work |
|------|-----------------|----------------|
| 목적 | 변경 **범위 최소화** | 데이터 흐름 **끝까지 점검** |
| 질문 | "이 줄이 요청에 필요한가?" | "이 데이터가 후속 화면에 어떻게 흐르나?" |
| 인접 코드 | 안 건드림 (개선 금지) | 데이터가 흐르면 후속 화면도 **고쳐야 함** |
| 충돌 해소 | 데이터 흐름상 후속 화면 수정이 **요청 달성에 필수**면 그건 "요청 범위 내" → connected 따라 수정. 단순 "보기 좋게" 인접 개선만 surgical 이 차단. |

요약: connected 가 "고쳐야 한다" 한 후속 화면은 surgical 위반 아님(요청 범위). surgical 이 막는 건 요청과 무관한 "겸사겸사" 개선뿐.

## 다른 게이트와의 구분

| 게이트 | 축 |
|--------|----|
| **surgical-change** (이 파일) | 변경 **범위** 최소화 — 요청한 것만, 인접 안 건드림 |
| `simplicity-first` | 변경 **방식** 최소화 — 적은 코드 / 추측기능 0 / 과설계 0 |
| `connected-work` | 데이터 흐름 6단계 (입력→BE→표시) |
| `polish-pass` | UI/UX 완료 기준 DoD 7항목 |
| `concise-response` | 답변 **길이** 최소화 (코드 아님) |

## 이중 방어

| 계층 | 도구 | 역할 |
|------|------|------|
| L1 — 규칙 | 이 파일 | Claude 자발적 — diff hunk 추적성 자문, 답변 검증 표현 강제 |
