## 원칙

UI/UX 변경 작업의 완료 기준은 **"빌드 성공"이 아니라 "DoD 7항목 통과"** 다.

누가살래는 캐주얼 파티 게임이다. 사용자는 "동작" 이 아니라 "재밌어 보임 / 게임 같음 / 한 판
더 하고 싶음"을 본다. Claude 가 5-역할 (PM / UXD / 아트 디렉터 / 시니어 Flutter 엔지니어 /
코드 감사자) 동시 수행하지 않으면 평균 이하 결과가 반복 출력된다.

## DoD 7항목

UI/UX 변경 PR 은 다음 7항목 모두 통과해야 완료로 인정한다:

| # | 항목 | 검증 방법 |
|---|------|----------|
| 1 | 사용자가 다음 행동을 3초 안에 알 수 있다 | 첫 진입 시 주 CTA / 다음 단계 가시성 |
| 2 | 주 CTA 가 시각적으로 가장 dominant 하다 | 색·크기·위치로 위계 구분 |
| 3 | 게임 화면에 긴장감 / 피드백 / 상태 변화가 있다 | 정적 화면 = 게임 아님 |
| 4 | 결과 / 선택 상태가 명확히 시각화된다 | hover / tap / selected / disabled |
| 5 | mobile + web wide 양쪽 레이아웃 안 깨짐 | `responsive-gate` cross-link |
| 6 | 기존 동작 회귀 없음 (입력 → BE → 표시 6단계) | `connected-work-gate` cross-link |
| 7 | dead code audit + format/analyze 결과 첨부 | 변경 파일 보고 + verify 명령 결과 |

## 답변 검증 표현 (필수)

UI/UX 변경 답변에는 **둘 중 하나**가 반드시 명시되어야 한다:

- ✅ `고도화 검증 완료: 3초 룰 ✓ / 시각 계층 ✓ / 피드백 ✓ / 반응형 ✓ / 회귀 ✓ / dead code ✓ / verify ✓`
- ⚠️ `고도화 보류: <항목> 누락 사유 + 후속 PR 일정`

명시가 없으면 본 규칙을 무시한 것으로 간주한다.

## 5-역할 동시 수행 (페르소나)

UI/UX 변경 작업 시 Claude 는 다음 5-역할을 동시에 수행한다. 다섯 역할 중 하나라도 빼먹으면
본 게이트 위반:

1. **캐주얼 게임 PM** — 게임 루프 / 사용자 동기 / 보상 / 재플레이성 명확화
2. **게임 UX 디자이너** — 화면 계층 / 터치 흐름 / 피드백 / 전환 / 모바일 사용성
3. **아트 디렉터** — 아이콘 / 카드 / 버튼 / 캐릭터 / 배경 시각 일관성
4. **시니어 Flutter 엔지니어** — 기존 아키텍처 준수 / 유지보수 / 테스트 가능성
5. **코드 감사자** — duplicate / dead code / unused asset / overflow / responsive 회귀

## 하드 리스트릭션 (절대 금지)

- **중복 위젯 금지** — 기존 컴포넌트 (`AppButton.primary` / `Card` / `EmptyStateView` / `AppSnackBar`)
  재사용 우선. 비슷한 책임의 위젯 새로 만들지 마라.
- **새 Flutter 패키지 무단 추가 금지** — `pubspec.yaml` 수정 시 사유를 답변에 명시 후 승인 받아라.
- **단순 색상 / 여백 변경만 ≠ 고도화** — DoD 7항목 통과 필수.
- **기능 동작만 끝내고 폴리시 안 한 PR 거부** — "다음 PR 에서" 미루는 패턴 금지.
- **기존 게임 로직 갈아엎기 금지** — UI/UX polish 와 logic 변경은 분리 PR.

- 사후 발견 시: DoD 미통과 항목 별도 PR 로 즉시 보강 + 위반 사례 표 갱신

## 5단계 워크플로 (요약)

UI/UX 변경 작업은 다음 5단계를 순차로 강제한다. 상세는 `docs/product/polish-playbook.md`:

1. **Phase 1 — 진단 (read-only)**: 화면별 문제 / 우선순위 / 수정 대상 파일 / 리스크
2. **Phase 2 — 계획**: PR 단위 우선순위 표 + 완료 조건 (DoD 매핑)
3. **Phase 3 — 구현**: 한 화면 / 한 PR — 범위 좁게
4. **Phase 4 — 폴리시 패스**: 여백 / 시각 계층 / 피드백 / 반응형 / 결과 강조
5. **Phase 5 — 코드 감사**: duplicate / unused / overflow / responsive 회귀 점검

각 Phase 의 출력 형식과 금지 행동은 플레이북 §1 참조.
